How to Make a Commit

Making a commit involves a few steps: making changes to your files, adding those changes to the staging area, and then committing them with a descriptive message. Below are common ways to do this using the command line (Git Bash, Terminal, etc.) and briefly, via GitHub Desktop and other GUIs.

Committing via the Command Line (Most Common)

This is the most direct way to interact with Git and provides the most control.

  1. 1. Make Changes

    First, modify your project files. This could be editing code, writing text, adding images, or anything else related to your project.

  2. 2. Check the Status

    Before staging, it's good practice to see what Git has noticed about your changes. Use the command:

    git status

    This will show you which files are modified, which are new (untracked), and which are staged.

  3. 3. Stage Your Changes

    Next, you need to "stage" the changes you want to include in your commit. This tells Git which specific modifications will be part of the next snapshot.

    • To stage a specific file: 
      git add filename.ext

      (Replace filename.ext with the actual name of your file)

    • To stage all changes in the current directory and its subdirectories: 
      git add .

      (The dot . represents the current directory)

    • To stage all changes in the entire repository, no matter where you are: 
      git add -A

      (Or git add --all)

    After running git add, you can run git status again to see that your files have moved to the "Changes to be committed" section.

  4. 4. Commit the Staged Changes

    Once your desired changes are staged, you commit them. This is where you provide a message explaining what you did.

    git commit -m "Your descriptive commit message here"

    Tips for good commit messages:

    • Keep the subject line (the part in quotes) concise (around 50 characters if possible).
    • Use the imperative mood (e.g., "Add feature X," "Fix bug Y," not "Added feature X" or "Fixing bug Y").
    • If you need more detail, you can omit the -m "message" part. This will open your configured text editor where you can write a longer message (a subject line, a blank line, then the body).

    Example:

    git commit -m "Add user login functionality"

  5. 5. Push Your Commits (Optional, for sharing/backup)

    If you're working with a remote repository (like one on GitHub), your commits are still only on your local machine. To share them, you need to "push" them:

    git push

    (If it's the first time pushing a new branch, you might need something like git push -u origin main or git push -u origin your-branch-name)

Committing via GitHub Desktop or other GUIs

Graphical User Interfaces (GUIs) like GitHub Desktop, Sourcetree, GitKraken, or VS Code's built-in Git tools provide a visual way to commit:

  1. View Changes: The GUI will typically show you a list of modified files.
  2. Stage Changes: You can usually select files (or parts of files) to stage by checking boxes or clicking buttons.
  3. Write Message & Commit: There will be a text box for your commit message and a "Commit" button.
  4. Push: A "Push" or "Sync" button will send your commits to the remote repository on GitHub.

While GUIs can be more intuitive initially, understanding the command-line workflow is beneficial for deeper Git comprehension and troubleshooting.
